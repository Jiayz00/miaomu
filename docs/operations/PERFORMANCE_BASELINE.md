# ShopXO 苗木站性能基线协议

## 当前结论

本文只固定 NFR-PERF-005 的测量协议，不包含真实性能结果。当前 Docker、Caddy、MySQL、HTTP 和七类业务场景均未执行，状态为 `NUR-OPS-001: not_run`。不得把静态合同测试、容器启动或单次页面访问表述为性能通过。

## 测量边界

性能请求必须经过实际用户入口的 Caddy，不得绕过 Caddy 直接请求 PHP-FPM。首轮服务器内无身份场景可在 `127.0.0.1:88` 测量；涉及登录、收藏、询价、管理后台或个人数据的场景，必须等待独立 TLS 域名、脱敏测试账号和相应业务功能验收完成。

基线只描述已实现功能。尚未实现、缺少安全测试账号或不具备前置数据的场景必须记录 `blocked` 或 `not_run`，不能以空接口、模拟成功响应或被拒绝请求代替。

## 固定环境指纹

每次基线报告必须记录：

- 测量时间、执行角色、L4 task ID 和 Git release SHA；
- 服务器 CPU、内存、存储、内核、架构和时区；
- Caddy v2.11.2 的镜像 ID、完整配置哈希、共享 Compose 哈希和当前路由；
- `miaomu-app` 镜像 ID、PHP 版本、扩展清单和 FPM 配置哈希；
- MySQL 镜像 ID、版本、关键配置和 schema/data fixture 版本；
- `deploy/compose.yaml`、stack policy、release manifest 和 database.php 模板哈希；
- app/db 资源限制、容器数量、health 状态、socket 路径/GID/mode；
- 数据集规模：商品、规格、分类、用户、收藏、询价、事件和日汇总行数；
- 缓存状态、后台任务状态、测试客户端机器和网络路径。

任何指纹变化都生成新的基线，不得与旧结果直接合并。

## 数据与安全

使用版本化、可重建且脱敏的性能 fixture。不得使用真实手机号、地址、询价正文、密码、token 或生产账号。写场景必须使用专用测试用户并在结束后按合同清理；清理失败视为测试失败。

原始请求与结果文件保存在仓库外的短期受控位置。Git 只提交脱敏摘要、环境指纹、统计值和结果哈希，不提交完整响应、cookie、Session、数据库备份或服务器原始日志。

## 统一测量方法

每个场景分别固定以下参数：

1. 明确 URL、HTTP 方法、请求体版本、身份类型和成功状态码。
2. 在正式采样前预热应用、数据库和必要缓存；预热请求不计入样本。
3. 记录并发级别、总样本数、持续时间、客户端超时、连接复用和思考时间。
4. 至少执行低、中、高三个预先锁定的并发档位；具体数值由 NUR-OPS-001 根据部署资源和安全上限写入合同，本文不臆造阈值。
5. 每个档位独立报告 P50、P95、最大值、吞吐量、错误数和错误率。
6. 同时记录 app/db/Caddy CPU、内存、磁盘 I/O、连接数、慢查询和错误日志的脱敏摘要。
7. 发生非预期 4xx/5xx、超时、数据污染、资源限制触发或共享站点回归时立即停止。

统计口径：

- 延迟从客户端开始发送请求到完整响应体接收结束；
- P50/P95 从成功样本的同一窗口计算，同时单独报告失败和超时，不得剔除后只报成功；
- 错误率 = 非预期失败请求数 / 总请求数；
- 预期权限拒绝不作为目标业务成功样本，也不能用于美化错误率；
- 报告必须保留每个档位的原始结果哈希和计算工具版本。

## 七类场景

| 场景 | 最小前置条件 | 关键断言 | 当前状态 |
| --- | --- | --- | --- |
| 商品列表 | 已上架脱敏商品、公开参考价可见 | 200、结果非空、价格字段一致 | `not_run` |
| 商品详情 | 固定商品与规格 fixture | 200、详情/规格参考价一致 | `not_run` |
| 收藏 | TLS、专用用户、收藏功能已验收 | 用户隔离、幂等结果、无询价副作用 | `not_run` |
| 询价 | TLS、专用用户、询价状态/快照已验收 | 写入成功、快照正确、无收藏副作用 | `not_run` |
| 行为上报 | 事件目录、visitor_id 与幂等规则已验收 | 主流程不被明显阻塞、无敏感字段 | `not_run` |
| 后台 30 日趋势 | 最小权限管理员、日汇总 fixture | 读取汇总、口径一致、不扫全量明细 | `not_run` |
| 数据导出 | 独立导出权限、脱敏规则和 fixture | 权限正确、文件完整、无敏感泄露 | `not_run` |

收藏、询价、行为、趋势或导出尚未实现时，对应责任任务必须先完成产品验收；NUR-OPS-001 只能记录 `blocked/not_run`，不能用占位接口补齐数字。

## 报告与后续基线

NUR-OPS-001 的真实性能报告应逐场景包含：环境指纹、fixture 版本、预热、并发、样本数、P50、P95、吞吐量、错误率、资源曲线摘要、异常和原始结果哈希。首轮结果用于固定后续回归基线，不在执行前虚构“通过”阈值。

后续版本只在相同环境指纹和数据规模下比较；若环境或 fixture 改变，应建立新基线并解释差异。任何真实性能结论在 NUR-OPS-001 实测前均为 `not_run`。
