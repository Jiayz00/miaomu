# 1. 文档说明

## 1.1 编写目的

本需求文档用于统一描述基于 ShopXO 二次开发的苗木商品信息平台需求，避免在开发过程中出现以下问题：

- 将项目误做成完整电商商城；
- 擅自加入供应商、多商户、分销、购物车、订单或支付；
- 将商品询价错误理解为“询问基础价格”；
- 将收藏错误设计成询价或购买的前置流程；
- 只统计累计浏览量，无法查看日、周、月趋势；
- 混淆浏览次数、访问人数、注册人数、收藏人数和询价人数；
- 对 ShopXO 核心代码进行无边界修改；
- 页面功能实现后缺少明确验收标准。

本文件是项目的业务与产品需求基线。后续数据库设计、接口设计、页面原型、开发任务和测试用例均应与本文保持一致。

## 1.2 需求优先级

| 优先级 | 含义 | 说明 |
|---|---|---|
| P0 | 必须实现 | 缺失则系统无法达到基本建设目标 |
| P1 | 重要实现 | 应在第一版或紧随其后的版本实现 |
| P2 | 增强功能 | 不阻塞第一版上线，可后续迭代 |
| PX | 明确排除 | 当前项目不允许实施 |

## 1.3 术语

| 术语 | 定义 |
|---|---|
| 商品 | 在平台展示的一种苗木品种或可独立维护的苗木销售单元 |
| 商品规格 | 会影响价格、库存或实际供应条件的组合，例如高度、胸径、冠幅、等级 |
| 商品参数 | 主要用于描述和筛选、不直接形成价格组合的属性 |
| 参考价格 | 管理员预先维护并在商品页面公开展示的价格或价格区间 |
| 收藏 | 用户保存感兴趣商品，以便后续快速找到和比较 |
| 询价 | 用户针对具体商品、规格、数量、地区和服务条件提交进一步采购咨询 |
| PV | 页面浏览次数，同一访客多次打开可累计多次 |
| UV | 独立访客数，在指定统计周期内按访客标识去重 |
| 会话 | 用户一次连续访问过程 |
| 事件 | 用户在页面产生的可统计操作，例如查看详情、播放视频、收藏、询价 |
| ShopXO核心 | 原仓库中通用控制器、服务、框架、系统配置及公共业务代码 |
| 苗木扩展模块 | 为本项目新增的询价、统计、价格历史及苗木特定配置能力 |

# 2. 项目背景与建设目标

## 2.1 项目背景

苗木商品与普通标准化零售商品不同，通常具有以下特点：

- 同一品种因高度、地径、胸径、冠幅、苗龄、等级等产生不同价格；
- 商品价值高度依赖图片、视频和现场状态；
- 实际采购价格可能受数量、库存、运输距离、装卸和栽植服务影响；
- 用户往往先查询、比较和保存商品，再根据具体需求发起询价；
- 管理员需要了解哪些商品被浏览、收藏、搜索和询价，以调整商品结构和展示内容。

因此，本项目需要建设一个以商品信息展示和客户需求收集为核心的平台，而不是传统在线下单商城。

## 2.2 总体目标

建设一个面向单一苗木经营主体的商品展示、用户查询、商品收藏、在线询价和运营数据分析平台，实现：

1. 苗木商品信息统一管理；
2. 商品分类、搜索和筛选；
3. 商品图片、视频、规格、参数和参考价格展示；
4. 用户注册、登录和个人中心；
5. 用户收藏感兴趣的商品；
6. 用户对具体商品提交询价；
7. 管理员回复并跟踪询价状态；
8. 统计网站、用户、商品、搜索、收藏和询价数据；
9. 展示总体和单商品趋势曲线；
10. 在尽量少修改 ShopXO 核心的前提下完成二次开发。

## 2.3 项目定位

本项目定位为：

> **单经营主体的苗木商品信息展示与客户询价管理平台。**

平台核心业务路径为：

```text
访客进入网站
→ 分类、搜索或筛选商品
→ 查看商品详情及公开参考价格
→ 注册或登录
→ 收藏感兴趣商品，或直接提交询价
→ 管理员查看并处理询价
→ 用户查看回复并开展后续线下沟通
```

收藏和询价为两个独立操作，不构成强制前后关系。

# 3. ShopXO 基础能力分析

## 3.1 仓库与技术基础

当前参考仓库为：

```text
gongfuxiang/shopxo
默认分支：master
```

仓库当前公开说明表明：

- 使用 MIT 开源协议；
- 支持商用和二次开发；
- 支持 PC、手机自适应和多种移动端形态；
- 支持插件扩展、可视化 DIY 和表单能力；
- 独立移动端代码位于 `shopxo-uniapp` 仓库。

当前 `composer.json` 显示的主要技术基础包括：

- PHP `>= 8.0`；
- ThinkPHP Framework；
- ThinkPHP ORM；
- ThinkPHP 多应用；
- ThinkPHP View；
- PhpSpreadsheet；
- PHPMailer；
- PSR-4 命名空间 `app\` 映射至 `app/`。

## 3.2 当前架构特点

ShopXO 当前采用多应用结构，核心目录职责可概括为：

```text
app/admin       管理员后台控制器、表单和视图
app/index       PC 用户端控制器和视图
app/api         API 和移动端接口
app/service     共享业务服务层
app/plugins     插件与扩展模块
config          配置与安装 SQL
public/static   前端静态资源
```

项目整体表现为“控制器调用服务层”的架构，商品新增、编辑、删除和状态变更主要由 `GoodsService` 处理。

## 3.3 可直接复用的 ShopXO 能力

### 3.3.1 商品管理

现有后台商品编辑流程已经支持：

- 商品分类；
- 商品标题和简介；
- 商品规格；
- 商品参数；
- 商品图片；
- 商品视频；
- PC 详情；
- 移动端详情；
- 使用指南；
- SEO；
- 上下架状态。

后台商品控制器 `app/admin/controller/Goods.php` 会加载分类、品牌、地区、商品参数和规格数据，并将保存操作交给 `GoodsService::GoodsSave()`。

### 3.3.2 商品保存

当前商品保存服务会处理：

- 商品主表；
- 商品分类关联；
- 商品规格；
- 商品规格价格；
- 商品相册；
- 移动端详情；
- 商品参数；
- 库存同步；
- 保存前后插件钩子；
- 数据库事务和异常回滚。

该能力可以作为苗木商品管理的主体，不应重新开发一套平行商品系统。

### 3.3.3 商品价格和规格

现有商品规格基础数据支持：

- 价格；
- 原价；
- 库存；
- 起购数量；
- 限购数量；
- 重量；
- 体积；
- 编码；
- 条码；
- 库存单位；
- 扩展数据。

现有商品服务会根据所有规格的最低价和最高价生成商品展示价格或价格区间，可用于苗木参考价格展示。

### 3.3.4 商品收藏

现有 `GoodsFavorService` 已支持：

- 登录用户收藏商品；
- 取消收藏；
- 判断是否已收藏；
- 按用户查询收藏列表；
- 统计商品收藏数量；
- 收藏列表关联当前商品标题、图片、价格、库存和上下架状态。

因此，收藏功能应在现有收藏服务基础上调整，不应新建第二套收藏主业务。

### 3.3.5 商品浏览

现有商品详情控制器已经：

- 增加商品访问次数；
- 保存用户商品浏览记录；
- 获取商品收藏数量；
- 获取商品图片、规格和参数；
- 调用商品详情页面插件钩子。

累计访问数据可以复用，但日趋势、UV、来源、会话和点击事件仍需新增统计模块。

### 3.3.6 插件与钩子

ShopXO 当前支持动态插件服务，例如：

```text
app\plugins\<插件标识>\service\BaseService
```

商品后台编辑和商品详情页面已经提供多个插件钩子，包括：

- 商品后台编辑页面钩子；
- 商品保存数据处理钩子；
- 商品保存事务前后钩子；
- 商品详情图片区域钩子；
- 商品详情价格区域钩子；
- 商品详情规格区域钩子；
- 商品购买导航区域钩子；
- 商品详情标签和内容区域钩子。

因此，本项目应优先采用独立苗木插件或扩展模块完成询价和统计能力。

## 3.4 需要新增或增强的能力

ShopXO 核心现有能力不足以完整满足以下需求：

- 结构化苗木询价；
- 询价回复和状态历史；
- 商品真实价格变更历史；
- 网站 PV、UV 和会话统计；
- 用户注册来源和注册漏斗；
- 商品每日访问趋势；
- 图片预览和视频播放统计；
- 搜索词和无结果搜索分析；
- 收藏新增、取消和净增长曲线；
- 询价点击、提交、回复和查看分析；
- 总体运营看板；
- 单商品数据分析；
- 数据导出和运营排行。

上述功能需要作为苗木扩展能力进行开发。

# 5. 用户角色与权限

## 5.1 角色定义

### R-01 游客

未登录访问网站的用户。

允许：

- 浏览首页；
- 查看分类；
- 搜索和筛选商品；
- 查看商品列表；
- 查看商品详情；
- 查看参考价格；
- 查看图片、视频、规格和参数；
- 进入注册和登录；
- 点击收藏或询价入口。

限制：

- 不可写入收藏；
- 不可提交询价；
- 不可查看用户中心。

### R-02 注册用户

完成注册和登录的普通用户。

允许：

- 使用全部游客功能；
- 修改个人资料；
- 收藏和取消收藏；
- 查看自己的收藏；
- 提交询价；
- 查看自己的询价和回复；
- 查看自己的浏览历史；
- 管理本人的联系方式。

限制：

- 不可查看其他用户收藏；
- 不可查看其他用户询价；
- 不可访问管理员后台；
- 不可修改商品和统计数据。

### R-03 商品管理员

负责商品分类和商品信息维护。

允许：

- 管理分类；
- 新增和编辑商品；
- 上传图片和视频；
- 维护规格、参数和参考价格；
- 上下架商品；
- 查看商品基础统计。

限制：

- 是否可查看完整用户联系方式由权限配置决定；
- 不默认拥有系统配置和全部询价导出权限。

### R-04 询价管理员

负责客户询价处理。

允许：

- 查看授权范围内询价；
- 回复询价；
- 修改询价状态；
- 查看商品和用户必要信息；
- 查看询价统计。

### R-05 数据查看人员

只读查看运营数据和报表，不修改商品或询价。

### R-06 超级管理员

拥有全部后台权限和角色配置权限。

# 6. 核心业务规则

## 6.1 商品与参考价格

### BR-PRICE-001

商品在上架前必须具有有效参考价格或有效价格区间。

### BR-PRICE-002

参考价格必须在商品列表和商品详情中直接展示，不得要求用户先提交询价。

### BR-PRICE-003

商品可以使用以下价格形式：

- 固定参考价格；
- 规格参考价格；
- 参考价格区间；
- 起始参考价格。

“面议”只允许作为特殊情况，不应成为大部分商品的默认价格。

### BR-PRICE-004

商品详情必须提示：

> 页面所示价格为参考价格，实际交易条件可能因规格、数量、库存、运输距离、装卸方式、栽植服务及市场变化而调整，最终以双方确认结果为准。

### BR-PRICE-005

管理员询价回复中的价格属于针对本次采购需求的业务回复，不得覆盖商品公开参考价格。

### BR-PRICE-006

商品价格发生修改时，应记录真实变更历史。新旧价格相同不得生成虚假变更记录。

## 6.2 收藏

### BR-FAV-001

收藏用于保存用户感兴趣的商品，主要目的是后续快速查找和比较。

### BR-FAV-002

收藏与询价相互独立：

- 可以只收藏；
- 可以不收藏直接询价；
- 可以询价后再收藏；
- 可以收藏多个商品，仅询价其中部分商品。

### BR-FAV-003

同一用户对同一商品只能存在一条有效收藏记录。

### BR-FAV-004

商品下架后，收藏记录应保留并标记“已下架”。

### BR-FAV-005

用户只能查看和管理自己的收藏记录。

## 6.3 询价

### BR-INQ-001

询价必须关联商品，可选择关联具体规格。

### BR-INQ-002

询价用于进一步确认：

- 具体规格；
- 采购数量；
- 当前库存；
- 批量价格；
- 运输和装卸；
- 栽植服务；
- 预计采购时间；
- 其他个性化条件。

### BR-INQ-003

提交询价不要求先收藏商品。

### BR-INQ-004

询价提交时必须保存必要商品快照，避免商品后续修改导致历史询价内容失真。

### BR-INQ-005

询价状态变化和管理员回复必须保留历史。

### BR-INQ-006

商品下架后，历史询价仍可查看。

### BR-INQ-007

系统必须限制短时间内相同用户对同一商品重复提交完全相同内容。

## 6.4 删除和历史数据

### BR-DATA-001

商品原则上采用逻辑删除，不应因为商品删除破坏收藏、询价和统计历史。

### BR-DATA-002

询价记录和回复历史不得被普通删除操作物理清除。

### BR-DATA-003

管理员关键操作必须记录操作人、时间和操作内容。

## 9.1 访客识别

### DATA-ID-002 会话

每次连续访问生成或维护 `session_id`。

会话超时时间应配置化，建议默认30分钟无操作后结束。

### DATA-ID-003 登录关联

用户注册或登录后，建立：

```text
visitor_id → user_id
```

关联，用于分析注册前后行为。

### DATA-ID-004 IP 使用

IP只能用于辅助安全和粗粒度地域分析，不得作为唯一UV识别依据。

IP应脱敏、截断或哈希处理。

## 9.4 核心指标定义

### METRIC-001 页面浏览量 PV

```text
统计周期内符合条件的 page_view 事件总次数
```

### METRIC-002 独立访客 UV

```text
统计周期内 visitor_id 去重数
```

登录用户切换设备可能产生多个 visitor_id，应在用户分析中另行统计独立登录用户。

### METRIC-003 商品点击率

```text
商品卡片点击人数 ÷ 商品卡片曝光人数 × 100%
```

同时保留按次数计算的辅助指标。

### METRIC-004 注册成功率

```text
注册成功人数 ÷ 注册提交人数 × 100%
```

### METRIC-005 收藏率

```text
收藏商品的独立用户数 ÷ 商品详情独立访客数 × 100%
```

### METRIC-006 询价点击率

```text
点击询价的独立用户数 ÷ 商品详情独立访客数 × 100%
```

### METRIC-007 询价提交率

```text
成功提交询价的独立用户数 ÷ 打开询价表单的独立用户数 × 100%
```

### METRIC-008 商品询价率

```text
成功提交询价的独立用户数 ÷ 商品详情独立访客数 × 100%
```

### METRIC-009 回复率

```text
已回复询价数 ÷ 有效询价总数 × 100%
```

### METRIC-010 回复查看率

```text
用户已查看回复的询价数 ÷ 已回复询价数 × 100%
```

### METRIC-011 平均首次回复时间

```text
首次管理员回复时间 - 询价提交时间
```

对有效询价求平均或中位数，后台应明确使用口径。

### METRIC-012 收藏净增长

```text
新增收藏数 - 取消收藏数
```

## 9.5 注册漏斗

第一版注册漏斗：

```text
注册页访问
→ 注册提交
→ 注册成功
→ 首次登录
→ 注册后查看商品
→ 注册后收藏或询价
```

收藏和询价是并列的后续行为，不是固定顺序。

## 9.6 商品行为路径

```text
商品卡片曝光
→ 商品卡片点击
→ 商品详情查看
→ 图片/视频交互
→ 收藏，或询价，或继续浏览
```

## 9.7 询价路径

```text
商品详情查看
→ 点击询价
→ 打开询价表单
→ 成功提交
→ 管理员回复
→ 用户查看回复
```

## 9.8 数据刷新

| 数据类型 | 建议刷新方式 |
|---|---|
| 累计浏览量 | 实时或近实时 |
| 收藏数量 | 实时 |
| 待处理询价 | 实时 |
| 当日事件趋势 | 近实时或按小时 |
| 日趋势 | 每日汇总 |
| 用户留存 | 每日计算 |
| 历史排行榜 | 汇总表查询 |

## 9.9 数据保留

建议默认：

- 询价及回复历史：长期保留；
- 商品价格历史：长期保留；
- 每日汇总数据：长期保留；
- 原始行为事件：保留180天，可配置；
- 会话明细：保留180天，可配置；
- 搜索日志：保留365天，可配置；
- 安全操作日志：不少于180天。

最终周期应根据服务器容量和隐私要求确认。

# 14. 关键验收场景

## AC-001 商品公开价格

1. 管理员新增商品并设置规格参考价格；
2. 商品上架；
3. 游客进入列表和详情；
4. 无需登录和询价即可看到参考价格；
5. 价格区间与规格数据一致。

## AC-002 收藏独立于询价

1. 用户登录；
2. 收藏商品；
3. 不提交询价；
4. 商品正常出现在“我的收藏”；
5. 用户取消收藏成功；
6. 整个过程不创建询价记录。

## AC-003 不收藏直接询价

1. 用户登录；
2. 未收藏商品；
3. 直接进入询价表单；
4. 成功提交；
5. 管理员后台看到询价；
6. 用户仍未被系统自动收藏该商品。

## AC-004 管理员回复

1. 管理员打开待处理询价；
2. 填写回复；
3. 状态变为已回复；
4. 用户查看；
5. 状态变为用户已查看；
6. 回复和状态历史完整保留。

## AC-005 商品下架历史保留

1. 用户收藏商品并提交询价；
2. 管理员下架商品；
3. 商品不再出现在公共列表；
4. 收藏列表显示已下架；
5. 用户仍可查看历史询价；
6. 后台历史统计不丢失。

## AC-006 商品趋势数据

1. 多个访客访问同一商品；
2. 同一访客重复访问；
3. 后台正确区分PV和UV；
4. 近7日曲线按日期展示；
5. 累计值与日汇总总和在统计口径下匹配。

## AC-007 注册来源

1. 游客从收藏按钮进入注册；
2. 注册成功；
3. 后台注册来源显示“收藏触发”；
4. 注册后收藏成功；
5. 用户与注册前 visitor_id 建立关联。

## AC-008 搜索无结果

1. 用户搜索不存在的苗木；
2. 页面正常提示无结果；
3. 后台记录搜索词和时间；
4. 无结果搜索排行可查看。

## AC-009 用户数据权限

1. 用户A创建收藏和询价；
2. 用户B尝试通过修改ID访问；
3. 系统拒绝；
4. 管理员授权角色可以查看；
5. 操作留有安全日志。

## AC-010 价格历史

1. 管理员将商品价格从100修改为120；
2. 系统记录真实变更；
3. 再次保存120不新增重复历史；
4. 当前价格为120；
5. 询价历史中的提交时价格快照不被覆盖。

# 17. 风险与注意事项

## 17.1 ShopXO 功能过多

风险：原系统大量商城入口影响用户体验。

措施：优先通过配置和导航精简，保留底层兼容能力。

## 17.2 核心升级冲突

风险：直接修改 `GoodsService` 等核心服务导致后续升级困难。

措施：优先使用插件钩子和独立模块；核心修改单独登记。

## 17.3 统计口径不一致

风险：浏览次数、访客人数、用户人数相互混淆。

措施：建立统一事件目录和指标定义，次数和人数分别存储。

## 17.4 数据量增长

风险：事件表增长影响数据库。

措施：使用按日汇总、索引、归档和保留周期。

## 17.5 商品参数不规范

风险：自由文本导致筛选无效。

措施：关键字段使用参数模板、单位规范和枚举选项。

## 17.6 视频和图片存储

风险：高清媒体占用空间。

措施：限制大小、生成缩略图、按需加载，后续评估对象存储。

## 17.7 客服能力边界

ShopXO README 描述了完整在线客服能力，但核心仓库中是否包含对应完整源码和授权范围需要实施时再次确认。本项目第一版以结构化询价为准，不将实时聊天作为P0依赖。

# 21. 结论

本项目应在 ShopXO 现有用户、商品、分类、规格、参数、媒体、收藏和权限体系上进行精简型二次开发。

核心结论如下：

1. 系统仅服务一个苗木经营主体；
2. 不建设供应商、多商户和分销；
3. 不建设购物车、在线订单和支付；
4. 商品在询价前公开展示参考价格；
5. 收藏用于保存感兴趣商品；
6. 收藏和询价相互独立；
7. 询价用于确认具体采购条件；
8. 商品、图片、视频、规格、参数和收藏优先复用 ShopXO；
9. 询价、价格历史和运营统计作为新增能力；
10. 管理员需要查看网站、注册、商品、搜索、收藏和询价趋势；
11. 每个商品需要独立数据和历史曲线；
12. 二次开发优先插件和钩子，尽量减少 ShopXO 核心修改；
13. 所有功能必须具备明确权限、测试和验收标准。

本文件可作为后续数据库设计、接口设计、页面原型、Harness任务合同、Codex开发计划和测试用例的统一输入。
