# NUR-OPS-001 需求摘录

## 关联需求

- `NFR-SEC-006`：密钥、数据库口令和生产配置不得进入 Git、日志或证据。
- `NFR-PERF-005`：部署环境确定后固定可复现的性能基线方法，并覆盖已实现和待实现入口。

## 任务路由

- PRIORITY: P0
- PHASE: 1
- RISK: L4
- TYPE: operations

本任务承接需求规格第一阶段“部署和确认 ShopXO 版本”。它现在包含服务器、共享 Caddy、数据库初始化、备份和回滚，因此必须使用 L4 合同与锁定的 `remote_execution`；不是普通本地 Compose 任务。

## 业务目标

将已审查的 ShopXO 苗木站提交部署到用户指定的个人服务器 `root@38.12.21.18:22` 的 `/root/jia/miaomu`，复用服务器现有 `jia-caddy`，只让 Caddy 在 `127.0.0.1:88` 提供回环验收入口。应用栈固定为两个长期服务 `app` 与 `db`，PHP 包含 `intl`/`Normalizer`，询价 HMAC 密钥只从仓库外 secret 注入。部署必须留下主机、镜像、配置、备份、回滚、HTTP 和性能基线证据；未执行的检查明确记为 `blocked` 或 `not_run`。

`NFR-SEC-006` 通过外部 SSH 文件引用、仓库外运行时配置、secret 元数据检查、输出脱敏和 Git 扫描落实。`NFR-PERF-005` 通过固定 Caddy/PHP/MySQL/数据集指纹、预热规则、并发、样本数、P50/P95 和错误率记录落实；收藏、询价、行为上报、趋势和导出等尚未具备数据或页面前置条件的场景不得填造结果。

## 明确不做

- 不在本任务实现或改写商品、分类、收藏、询价、统计、主题或 ShopXO 核心业务逻辑。
- 不新增或启用供应商、多商户、分销、购物车、订单、支付、退款和售后链路。
- 不访问生产环境、生产数据库或生产密钥，不发布正式公网入口，不创建 Nginx 或第二个长期 Caddy。
- 不使用原始 `ssh`、`scp`、`curl`，不在服务器现场编辑文件；所有远程动作只能由项目 broker 按本合同结构化 argv 执行。
- 不修改用户级/全局 Codex 配置，不读取或输出 `id_ed25519`、`known_hosts` 或任何应用 secret 内容。
- 不修改 `config/shopxo.sql`，也不新增或改写 nursery 迁移实现。固定上游安装基线之后，部署必须执行并验收已批准的 NUR-FEAT-002/003/004 catalog、favorite、inquiry v1 前向迁移；本任务只负责隔离测试库中的编排、备份、验证和失败回滚，不触碰生产或未知既有数据。

## 开放决策

本任务的两个 requirement 没有关联 `.harness/requirements-decisions.json` 中的开放产品决策；`decision_ids` 保持为空。以下是执行前提而不是产品口径决策，必须在远程动作前用只读证据确认：

1. 目标主机仍是用户授权的个人/测试服务器，未承载需要保留的生产数据。
2. 已有 `jia-caddy`、Beszel、80/443 路由和共享卷健康，端口 88 可供回环站点使用。
3. 外部 `id_ed25519` 与 `known_hosts` 文件存在且主机指纹精确匹配合同；不把文件内容复制进仓库。
4. 发布提交已完成独立计划/合并/发布审查，并由 release seal 锁定；自动审批由不同 Codex 角色执行，不要求用户逐步人工确认。
