# NUR-OPS-001 需求摘录

## 关联需求

- `NFR-SEC-006`
- `NFR-PERF-005`

## 任务路由

- PRIORITY: P0
- PHASE: 1

需求规格第 15.1 节把“部署和确认 ShopXO 版本”列为第一阶段第一项；后续 44 个 P0 功能都依赖完整源码、可执行工具链和测试环境，因此优先级为 P0、阶段为 1。`requirement-routing.md` 将框架/核心基础设施与生产发布归为 L4；本任务虽然只允许非生产测试服务器，但涉及部署、备份和回滚基础，按更严格的 L4 处理。

## 业务目标

`NFR-SEC-006`（需求规格第 2035-2037 行）要求数据库密码、短信密钥、邮件密码和生产配置不得进入 Git。任务将通过仓库外配置、权限限制、敏感信息扫描和脱敏证据落实这一要求；用户在对话中提供的生图凭据不得写入任何项目文件、日志或提交。

`NFR-PERF-005`（第 2057-2067 行）要求在部署环境确定后固定初始性能验收基准，建议覆盖商品列表、详情、收藏、询价、行为上报、后台 30 日趋势和导出。本任务先固定运行环境、测试入口、测量方法与预算记录位置；尚未实现的业务入口只能登记为后续相应任务的必测项，不能伪造性能结果。

需求规格第 2086 行要求代码和文档进入版本控制，第 2088-2099 行要求至少备份数据库、商品图片/视频、系统配置、插件代码和上传资源，并在数据库变更前备份及提供回滚说明。第 2206 行要求先部署和确认 ShopXO 版本。本任务把这些无独立编号的约束作为 `NFR-SEC-006` 与 `NFR-PERF-005` 的执行上下文，不扩充合同 requirement_ids。

## 明确不做

- 不实现商品、用户、收藏、询价、分析、看板或苗木主题，不修改任何业务行为。
- 不删除或启用供应商、多商户、分销、购物车、订单、支付、退款售后等 PX 能力；关闭这些能力属于后续独立任务。
- 不访问生产环境或生产数据，不自动发布生产；`38.12.21.18` 必须先由项目负责人确认为测试服务器。
- 不关闭火绒、不增加全局排除项、不修改用户级 Codex/SSH 配置，也不把 SSH 私钥或 API Key 复制进仓库。
- 不修改数据库结构。测试库如需初始化，仅使用固定上游 `config/shopxo.sql`，后续苗木迁移由独立数据任务管理。

## 开放决策

当前合同需求不关联 `.harness/requirements-decisions.json` 中的八项开放产品决策。完整源码已恢复，baseline-only bootstrap 已提交，`source-check` 已通过。仍有三个实施前提需要关闭，但它们不是产品口径决策：

1. `38.12.21.18:22` 已恢复 TCP/SSH 可达；远端写操作前仍须复核该主机无生产数据，并且不得影响现有 Caddy/Beszel 服务。
2. 指定不同于 `Codex` 和 `Jiayz00` 的真实 `release_approver`。
3. 已确认采用隔离的 Docker Compose 栈；实施时必须固定镜像摘要、资源限额、健康检查和备份位置。
