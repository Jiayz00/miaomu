# ShopXO 苗木项目宪章

## 1. 适用范围

本宪章约束当前仓库中的需求分析、设计、编码、数据变更、测试、审查和发布。项目目标是单一苗木经营主体的网站，不是通用多商户商城。

## 2. 冲突优先级

系统、开发者和用户在当前任务中的明确指令始终高于仓库文件。若用户要求改变既有产品规则，应先记录需求变更或决策，再更新 Harness，而不是让仓库规则假装覆盖用户授权。

在仓库内部的冲突顺序为：

1. 本宪章。
2. 已批准的需求决策。
3. `ShopXO苗木平台需求规格说明书_V1.0.md` 中的明确需求与业务规则。
4. 安全、数据与 ShopXO 边界规则。
5. 当前任务合同和获批计划。
6. 临时实现说明。

低优先级内容不得放宽高优先级约束。需求文档内部冲突必须进入 `.harness/requirements-decisions.json`，不得由实现者自行解释后编码。

## 3. 业务不变量

- 公开参考价格在商品列表和详情可见，询价不是看价门槛。
- 收藏与询价彼此独立，不得互相自动创建记录。
- 用户数据按用户隔离，管理员访问受角色和审计控制。
- 商品下架、逻辑删除和后续编辑不得破坏历史收藏、询价快照、回复或统计。
- 询价必须关联商品并保存必要快照，状态与回复历史只追加、不覆盖。
- 统计事件使用统一目录；PV、UV、用户数、次数分别定义和验证。
- 排除供应商、多商户、分销、购物车、在线订单、支付、退款售后等业务链路。
- ShopXO 中已有但排除的代码优先通过配置、菜单、模板入口和权限关闭，不做无分析的物理删除。

## 4. 工程不变量

- 业务代码修改必须有唯一任务合同、需求编号、明确范围和可验证验收项。
- 修改顺序为配置、现有服务、插件钩子、`nursery` 插件、独立模块、小范围核心适配。
- 数据库变更必须有版本化正向迁移、回滚、重复执行策略、测试数据与备份说明；`config/shopxo.sql` 不能作为既有站点的唯一升级脚本。
- 核心修改必须登记原因、替代方案、上游差异、升级风险和回滚方式。
- 测试结论必须来自实际命令、退出码和输出；缺失工具统一记为 `blocked`，未执行项不得进入通过证据。
- 仓库根及其祖先、必需源码、Harness 控制面、任务、状态、证据、报告和测试路径不得经过符号链接、Windows 目录联接或未知 reparse point；Python 3.11+ 必须失败关闭。测试输出与超时上限由代码固定，配置不得放宽。
- 不读取或记录密钥内容，不访问合同外数据库；远程或发布动作只允许在用户明确授权的 L4 operations 任务中，经项目 broker 严格按 `remote_execution` 锁定主机指纹、外部凭据引用、受管根和精确动作执行。mutating 动作还必须锁定已批准的干净 Git release commit。
- L3/L4、高风险权限、数据、统计和核心修改不得由实现角色自行批准。owner、reviewer、release_approver 可由不同 Codex 代理承担；自动审批必须在任务 v2 合同中预先锁定 implementation/plan/merge/release canonical agent task，审批 thread 必须不同于 implementation thread，L4 发布代理还必须独立于其余角色，并引用多轮审查、验证、备份和回滚证据。
- Codex agent task、`CODEX_THREAD_ID`、审查制品和 Git 记录只是项目内自声明审计上下文，不是密码学身份或远程授权凭据；实现代理不得据此自批。首次进入实现前，制品或锁定上下文变化会使 plan 审批失效；任务已实际进入 `implementing` 后，只有四份计划制品的漂移可改为显式 warning 并要求重新 `preflight`，由必需的独立 merge 代码/功能审查锁定当前内容。任务合同、执行策略、需求决策、远程目标或动作变化仍使 plan 审批失效；L4 release 审查及远程安全门不变。

## 5. 项目级隔离

Codex 扩展只允许存在于本仓库：

- `.codex/config.toml` 和 `.codex/hooks.json`；
- `.agents/skills/`；
- `.harness/` 与 `scripts/harness*`。

不得通过本项目任务修改用户级或系统级 Codex 配置、skill、plugin、MCP、hook 或 marketplace。

## 6. 事实与证据

需求文档中的 ShopXO 文件、钩子、表结构和能力均需在实际接入版本中重新核验。基线状态只能使用 `confirmed`、`unknown`、`not_available` 或 `blocked`；不得把推测标成已确认。

## 7. Bootstrap 例外

首次搭建、修复和验证 Harness 可不创建业务任务，但只能修改 Harness 自身、需求/架构文档和项目级 Codex 文件。ShopXO 业务代码不在例外范围内。
